# SETUP-CODE FORMAT v1 — the provisioning contract

> **Status:** `binding` — published 2026-07-11 (slice (e) of the ORDER 000
> follow-up). Machine twin: [`idle_engine/provisioning.py`](../idle_engine/provisioning.py)
> (`encode_setup` / `decode_setup` / `validate_against_catalog`). The worked
> examples below are generated by the REAL encoder and kept honest by
> `tests/test_provisioning.py::test_doc_examples_decode_and_reencode_exactly` —
> edit either side alone and the suite goes red.

A **setup code** is one short paste-able string that provisions an idle-game
instance for a Discord server: which theme pack to load, and which engine
features to enable. A server admin (or the websites lane acting for them)
pastes it once; the plugin decodes it at its trust boundary. Typos and
tampering fail loud (checksum), and the format is versioned so a future v2
can exist without breaking v1 consumers.

## Consumers

- **Encoder — the websites lane** (superbot-web): builds codes from the
  theme/feature choices an admin makes on the website *before the bot is
  invited* (README: the catalog is "eventually choosable on the website").
  Encode with `idle_engine.provisioning.encode_setup`, or reimplement from
  the grammar below — the doc-honesty test pins the byte-level behavior.
- **Decoder — the superbot-next plugin** (this engine, plugin-native): calls
  `decode_setup` on the pasted string (pure, no I/O — safe at the trust
  boundary), then `validate_against_catalog` against its shipped `themes/`
  directory at instance-load time.

Both sides speak THIS document. The format is the contract; the Python
module is its reference implementation.

## Code shape

```
IDLE1-0XJPESSDCSGQ4VBYVM
└─┬─┘ └───────┬────────┘
prefix        body: Crockford base32 of payload ++ checksum
```

## Grammar (byte-by-byte)

```
code     := prefix body
prefix   := "IDLE" version "-"          ; ASCII; version = decimal integer, no leading zeros
                                        ; v1 codes carry exactly "IDLE1-"
body     := crockford32(payload ++ checksum)
payload  := flags theme_id
flags    := 1 byte                      ; feature bitmask, see field table
theme_id := 1..32 bytes ASCII           ; schema-v1 slug: ^[a-z0-9]+(-[a-z0-9]+)*$
checksum := 2 bytes                     ; CRC-32 (zlib) of payload, & 0xFFFF, big-endian
```

"No leading zeros" means a zero-led version string (`IDLE01-`, `IDLE010-`)
is not a well-formed prefix at all — `MalformedCodeError`, never parsed as a
version — while the single digit `0` IS the well-formed integer zero and
fails as `UnknownVersionError` like any other version ≠ 1.

`crockford32` is Crockford base32: most-significant-bit-first 5-bit groups
over the alphabet `0123456789ABCDEFGHJKMNPQRSTVWXYZ` (no `I`, `L`, `O`,
`U`), final partial group zero-padded on the right. Encoding is canonical
(uppercase, no separators, zero padding bits) so equal configs produce
byte-identical codes. Decoding is **tolerant**: case-insensitive, internal
hyphens ignored, look-alikes folded (`O`→`0`, `I`→`1`, `L`→`1`); `U` and
anything else outside the alphabet is malformed, and non-zero padding bits
are malformed (exactly one accepted spelling per payload modulo the
tolerated substitutions).

### Worked bytes — `egg-farm`, all features on

| Step | Bytes (hex) | Meaning |
|---|---|---|
| flags | `07` | `0b00000111` = offline_progress + upgrades + prestige |
| theme_id | `65 67 67 2d 66 61 72 6d` | ASCII `egg-farm` |
| payload | `07 65 67 67 2d 66 61 72 6d` | flags ++ theme_id |
| checksum | `7e dd` | `crc32(payload) & 0xFFFF = 0x7edd`, big-endian |
| body input | `07 65 67 67 2d 66 61 72 6d 7e dd` | 11 bytes → 88 bits → 18 base32 chars (2 zero pad bits) |
| code | — | `IDLE1-0XJPESSDCSGQ4VBYVM` |

## Fields

| Field | Where | Type | Constraint |
|---|---|---|---|
| version | prefix | decimal int | v1 codes: exactly `1`; any other value → `UnknownVersionError` |
| `flags` | payload byte 0 | bitmask | bits 3–7 MUST be 0 in v1; set → `UnknownFeatureError` |
| `flags` bit 0 | payload | bool | `offline_progress` — closed-form offline credit (`idle_engine/engine.py`) |
| `flags` bit 1 | payload | bool | `upgrades` — the upgrade shop (`idle_engine/upgrades.py`) |
| `flags` bit 2 | payload | bool | `prestige` — the reset track (`idle_engine/prestige.py`) |
| `theme_id` | payload bytes 1.. | slug | 1–32 chars, `^[a-z0-9]+(-[a-z0-9]+)*$`; a `theme.id` from the catalog ([`theme-schema.md`](theme-schema.md)) |
| checksum | last 2 body bytes | uint16 BE | `crc32(payload) & 0xFFFF`; mismatch → `ChecksumError` |

Every v1 flag maps to a mechanic the engine ships today — there are no
reserved-for-later feature bits. In-code source of truth:
`idle_engine.provisioning.FEATURE_BITS`.

## Config semantics

Decoding yields a `SetupConfig(theme_id, offline_progress, upgrades,
prestige)`. `decode_setup` validates **shape only** (it is pure — usable at
the trust boundary with no filesystem); resolving the config against a
shipped catalog is the separate `validate_against_catalog(config,
themes_dir)` step, which:

- resolves `theme_id` to a loaded pack, else `UnknownThemeError`;
- rejects a config whose enabled feature has no skin in the resolved pack
  (`upgrades` on with no `upgrades` block, or `prestige` on with no
  `prestige` block) with `FeatureUnavailableError` — the engine would have
  nothing to render. `offline_progress` is pure mechanics and needs no skin.
  A pack *may* skin more than the config enables: toggles turn features off,
  packs never turn them on.

## Versioning policy

- **v1 is FROZEN once merged.** The grammar, field table, alphabet,
  checksum, and flag-bit assignments above never change under the `IDLE1-`
  prefix.
- Anything more than this document's prose — a new field, a new flag bit, a
  different payload layout — is **v2**: new `IDLE2-` prefix, new section
  published here first, decoders keep accepting v1 forever.
- v1 decoders seeing a future version fail with `UnknownVersionError`
  (distinct and actionable: "update the plugin"), never misparse. Unknown
  v1 flag bits likewise fail (`UnknownFeatureError`) rather than being
  ignored, so a v2-built config can never silently degrade on a v1 decoder.

## Error taxonomy

All errors subclass `SetupCodeError(ValueError)`; each condition below is a
distinct class, tested distinctly:

| Error | Raised by | Condition |
|---|---|---|
| `InvalidConfigError` | `encode_setup` | config not representable (theme id not a slug) |
| `MalformedCodeError` | `decode_setup` | no `IDLE<n>-` prefix (incl. leading-zero version, e.g. `IDLE01-`), illegal body char (incl. `U`), empty/truncated body, non-canonical padding, payload theme id not a slug / not ASCII |
| `UnknownVersionError` | `decode_setup` | prefix version ≠ 1 |
| `ChecksumError` | `decode_setup` | body decodes, checksum disagrees (typo/tampering) — checked before any payload semantics |
| `UnknownFeatureError` | `decode_setup` | flags bits 3–7 set |
| `UnknownThemeError` | `validate_against_catalog` | theme id not in the shipped catalog |
| `FeatureUnavailableError` | `validate_against_catalog` | enabled feature has no skin in the resolved pack |

## Cross-language test vectors (machine-readable)

Reimplementing either side of this contract in another stack? Consume
[`tests/vectors/setup-codes.v1.json`](../tests/vectors/setup-codes.v1.json)
— a versioned, language-agnostic vector file generated by the REAL codec:

- **valid** vectors: config → canonical code for every shipped pack ×
  a bounded feature-combo set, each with layer-by-layer intermediates
  (flags byte, payload hex, CRC-16 hex) so a foreign implementation can
  debug encode/checksum/base32 separately;
- **tolerance** vectors: case / internal-hyphen / whitespace /
  look-alike spellings that must decode to the same config (canonical
  form included);
- **error** vectors: malformed inputs, each with the expected error
  class name from the taxonomy above (decode-side classes only —
  catalog resolution needs a `themes/` directory, not a vector file).

Regenerate with `python3 tools/gen_setup_vectors.py`.
`tests/test_setup_vectors.py` holds the file honest both ways: the
committed JSON must be byte-identical to a fresh regeneration
(regenerate-or-red), and every vector is replayed through the live
codec — so both repos' suites can consume the same committed file and
contract drift reds a test run instead of a live decode.
