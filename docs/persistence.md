# SAVE FORMAT v2 — the GameState persistence contract

> **Status:** `binding` — v1 published 2026-07-11; v2 published the same
> day by the achievements slice (added `milestones`, with the v1→v2
> migration in the same PR). Machine twin:
> [`idle_engine/persistence.py`](../idle_engine/persistence.py)
> (`dump_state` / `load_state` / `_MIGRATIONS`). The worked examples below are
> generated by the REAL encoder and kept honest by
> `tests/test_persistence.py::test_doc_worked_example_dumps_exactly` — edit
> either side alone and the suite goes red.

A **save** is one canonical JSON string that captures a
[`GameState`](../idle_engine/state.py) exactly. The engine's states were
always trivially serializable by design (plain dicts and ints, integer-exact
math); this contract makes that a versioned, validated seam instead of an
accident. Whatever runtime eventually hosts the engine — the superbot-next
plugin ([PLUG-001 un-parked 2026-07-12; contract verified at superbot-next
`docs/game-plugin-contract.md` @ `d3dba9b`](plugin-adapter-scoping.md)) or
anything else — persists and restores game instances through these two
functions and never needs to know the state's field-by-field shape. Under that
contract persistence is **host-owned** (a plugin's `stores` facet is refused at
the gate), so these saves ride the host store rather than a plugin-shipped
backend.

## Scope: strings in, strings out — storage is the runtime's job

`dump_state(state) -> str` and `load_state(text) -> GameState` are pure:
stdlib only, no I/O, no clock, no randomness. **Where save strings live —
file, database, key-value store, a message the bot edits — is the storage
BACKEND, and the backend is the runtime's concern, not the engine's.** The
engine defines what a save *is*; the plugin decides where it *sits*. That is
the same seam split as [provisioning](provisioning.md) (`decode_setup` is
pure; catalog resolution is a separate step) and it is why there is no
checksum here: saves are machine-carried, never typed by hand, so integrity
and durability belong to the backend (checksums, backups, transactions),
while *validity* belongs to this contract.

## Canonical form

`dump_state` emits exactly one spelling per state:

- **JSON object, keys sorted lexicographically** at every level;
- **no whitespace** (separators `","` / `":"`);
- **ASCII-escaped** (`ensure_ascii`) — the string is plain ASCII bytes on
  every platform;
- **all eight v2 fields always present**, including empty mappings;
- **every quantity a JSON integer** — floats appear NOWHERE in a save. The
  engine's arithmetic is integer-exact (see
  [`architecture.md`](architecture.md) § Invariants); the save format
  refuses `1.0`, `true`, and `"1"` alike so no platform can smuggle float
  drift or type coercion back in. Python's unbounded ints serialize exactly
  at any magnitude (round-trip is tested at 10^3000).

Same state → byte-identical string: saves can be compared, hashed, deduped
and diffed as strings. `load_state` accepts any JSON spelling of a valid
document (whitespace and key order are not content) and canonicalization is
idempotent: `load(dump(s)) == s` for every valid state, and
`dump(load(t)) == t` for canonical `t`.

### Worked examples (generated by the real encoder — tested literals)

The empty state (`GameState()`):

```json
{"balances":{},"last_seen":0,"lifetime":{},"milestones":{},"owned":{},"prestige":{},"state_version":2,"upgrades":{}}
```

A populated state — `balances {"cur-a": 42}`, `owned {"gen-a": 3}`,
`last_seen 1700000000`, `upgrades {"up-a": 2}`, `lifetime {"cur-a": 999}`,
`prestige {"meta": 1}`, `milestones {"owned-1": 1}`:

```json
{"balances":{"cur-a":42},"last_seen":1700000000,"lifetime":{"cur-a":999},"milestones":{"owned-1":1},"owned":{"gen-a":3},"prestige":{"meta":1},"state_version":2,"upgrades":{"up-a":2}}
```

## Fields

| Field | Type | Constraint | Meaning (mirrors `GameState`) |
|---|---|---|---|
| `state_version` | int | exactly `2` for v2 documents | format version; drives migration dispatch |
| `balances` | object | str keys → int ≥ 0 | currency id → units held this run |
| `owned` | object | str keys → int ≥ 0 | generator spec id → count owned |
| `last_seen` | int | ≥ 0 | unix seconds up to which production is credited |
| `upgrades` | object | str keys → int ≥ 0 | upgrade spec id → level purchased |
| `lifetime` | object | str keys → int ≥ 0 | currency id → units EARNED this run (prestige driver) |
| `prestige` | object | str keys → int ≥ 0 | prestige currency id → units held (survives resets) |
| `milestones` | object | str keys → int ≥ 0 | milestone spec_id → `1` once earned (survives resets; `0` counts as unearned) — added in v2, [`design/achievements-v0.md`](design/achievements-v0.md) |

Mapping keys are the engine's opaque ids ([theme-schema.md](theme-schema.md)
slugs for shipped packs, arbitrary strings for hand-built specs) — the save
format does not constrain them beyond "JSON string". Entries with value `0`
are preserved, not pruned: `{"a": 0}` and `{}` are different states and
round-trip as such.

## Version policy

- **Each version is FROZEN once merged.** The field table above never
  changes under `"state_version": 2`; the v1 table (this table minus the
  `milestones` row, `state_version` exactly `1`) stays frozen forever as
  the input contract of the v1→v2 migration.
- **Any change to the document shape is a version bump routed through the
  migration registry** — `_MIGRATIONS: dict[int, callable]` in
  `persistence.py`, mapping source version *n* to a function that returns
  the document at *n + 1* (bumping `state_version` itself). The loader
  walks old documents up the chain step by step, then applies current-
  version validation to the result, so migrated saves face exactly the
  same strictness as native ones.
- **The registry carries its first REAL entry at v2**: `1 → 2` adds
  `milestones: {}` (achievements did not exist before v2, so every legacy
  save migrates with an empty earned set) and refuses a v1 document that
  already carries a `milestones` key (never a valid v1 save — refused
  loudly rather than silently wiped). The multi-step dispatch remains
  proven by a synthetic v0→v1 migration in `tests/test_persistence.py`.
  Policy, honored by this bump: a format change ships IN THE SAME PR as
  the migration covering the previous version; a bump without a migration
  is a broken contract, not a shortcut.
- **Future versions fail loud**: a `state_version` above what the loader
  speaks (or below with no registered chain) is `UnknownVersionError` —
  distinct and actionable ("update the plugin"), never misread as the
  current version.

## Error taxonomy

All errors subclass `SaveError(ValueError)`; each condition below is a
distinct class, tested distinctly (the [provisioning](provisioning.md)
taxonomy pattern):

| Error | Raised by | Condition |
|---|---|---|
| `InvalidStateError` | `dump_state` | input is not a serializable GameState (non-int/negative quantities, non-string ids) — dumping refuses what loading would refuse |
| `MalformedSaveError` | `load_state` | not a string, not parseable JSON, non-JSON constants (`NaN`/`Infinity`), top level not an object; also a migration returning a non-document |
| `UnknownVersionError` | `load_state` | integer version with no path to v1 (from the future, or old with no registered migration); also a migration mis-bumping |
| `FieldSetError` | `load_state` | `state_version` missing, a current-version field missing, or unknown extra fields — reported as a set, every offender named at once; also a v1 document smuggling the v2-only `milestones` field |
| `FieldTypeError` | `load_state` | a field or mapping entry has the wrong JSON type; floats and booleans are wrong types everywhere |
| `NegativeValueError` | `load_state` | a well-typed quantity below zero |

Nothing degrades silently: a save either loads into a GameState that
round-trips exactly, or raises one of the classes above. The tamper/fuzz
suite (`tests/test_persistence.py`) pins that dichotomy over seeded
corruptions of real saves.

## Golden save corpus (cross-version test vectors)

[`tests/vectors/saves.v2.json`](../tests/vectors/saves.v2.json) is the
machine-readable golden corpus for this contract, generated from the REAL
codec by [`tools/gen_save_vectors.py`](../tools/gen_save_vectors.py) (the
setup-code vector pattern, applied to saves). Three categories: **golden
v2** — representative states (fresh, mid-run, post-prestige, milestones
earned, ≥ 10^300 magnitudes, explicit-zero entries, non-ASCII ids) with
their canonical strings; **golden v1 migration** — frozen-field-set legacy
v1 documents with their byte-exact expected v1→v2 results; **errors** —
inputs naming their expected taxonomy class. `tests/test_save_vectors.py`
keeps it honest both ways: the committed file must be byte-identical to a
fresh regeneration (regenerate-or-red), and every vector replays through
the live `dump_state`/`load_state`. Regenerate after any deliberate format
change with:

```
python3 tools/gen_save_vectors.py
```

**Policy (binding, extends § Version policy): any future `state_version`
bump MUST extend this corpus in the same PR** — the existing golden v1/v2
documents gain their expected migrated results under the new version, so
what old saves become is always pinned byte-exactly before the bump can
merge; the regenerate-or-red test makes skipping this loud, never silent.

## Consumers

- **The future plugin/runtime**: `dump_state` after applying player
  actions, store the string in its backend of choice; `load_state` at
  instance wake-up, then hand the state to
  [`engine.apply_offline_progress`](../idle_engine/engine.py). Loading is
  strict enough to sit at a trust boundary.
- **Tooling/tests**: canonical strings are the repo's determinism currency —
  byte-comparing trajectories (`tests/test_properties.py`) and future
  simulator runs can use `dump_state` directly instead of ad-hoc
  serializers.
