## What happened

Shipped slice (e)'s parked idea: the machine-readable cross-language
test-vector file so the websites lane can build a byte-compatible
setup-code encoder. One build PR after a control fast-lane claim (PR #23,
`control/claims/setup-code-test-vectors.md`, merged then removed here).

1. **`tests/vectors/setup-codes.v1.json`** — versioned (`format_version`
   1), language-agnostic, generated BY the real codec, 90 vectors:
   - 30 **valid**: 6 shipped packs × a bounded combo set (all-on,
     all-off, each single feature — representative, not the exhaustive
     2^n product), each with layer intermediates (`flags_byte`,
     `payload_hex`, `crc16_hex`) so a foreign implementation debugs
     encode/checksum/base32 separately;
   - 37 **tolerance**: lowercase, hyphen-grouped, surrounding
     whitespace, O→0 folds (all-on bodies), I/L→1 folds (all-off
     bodies — the all-on codes contain no `1`, found while generating),
     kitchen-sink combined variants; each carries the canonical form;
   - 23 **errors**: malformed prefix/tag/version-charset, empty/short
     body, `U` + junk chars, non-canonical padding, unknown versions
     (0/2/42), checksum tamper in BOTH payload and checksum regions,
     unknown flag bits 3 and 7, six bad payload-slug shapes — each
     naming its expected taxonomy class (decode-side classes only;
     catalog resolution needs `themes/`, not a vector file).
2. **`tools/gen_setup_vectors.py`** — deterministic generator: no
   clock/randomness; theme ids come from gate-pinned `themes/*.yaml`
   stems (a 7th pack reds the parity test until regenerated); every
   vector sanity-checked against the live codec before emission.
3. **`tests/test_setup_vectors.py`** (156 tests, suite 216 → 372) —
   regenerate-or-red: committed JSON must be byte-identical to an
   in-memory `render()`; consumer replay: every valid vector encodes
   AND decodes to its stated values with intermediates recomputed
   independently via `zlib.crc32`, tolerance vectors decode to the same
   config, error vectors raise EXACTLY their stated class (not a
   subclass); structural pins (counts, catalog coverage, combo set,
   taxonomy closure).
4. **`docs/provisioning.md`** — new "Cross-language test vectors"
   section (file pointer + regeneration command); all pinned literals
   untouched, doc-honesty tests stay green.

⚑ **Found, reported, not fixed** (contract ambiguity, not redesigned
here): `decode_setup` accepts leading-zero version prefixes —
`IDLE01-<body>` parses as v1 (`_PREFIX_RE` + `int()`), while the doc
grammar says "version = decimal integer, no leading zeros". Left OUT of
the vector file so neither behavior gets enshrined; needs a one-line
ruling (tighten the regex or loosen the grammar prose).

Verify: `python3 -m pytest -q` → 372 passed; `python3 bootstrap.py
